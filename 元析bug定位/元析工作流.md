这个工作流本质上是：

用户给一个需求，元析把它一步步变成：明确需求 → 明确仓库 → 明确方案 → 拆任务 → 自动编码 → 自动 review → 自动部署 → 自动集成测试。

它不是单纯“写代码”，而是一个完整的 AI 后端研发流水线。文档里也明确说，元析的目标是覆盖“需求解析 → 需求澄清 → 原型确认 → 技术方案 → 编码 TDD → 代码评审 → 自动部署 → 集成测试”的研发闭环。

一、第一段：把“用户需求”变成 AI 能处理的标准输入
用户输入需求
 ↓
yx-fsd-req-km-extractor

用户一开始可能给的是：

FSD 链接、KM 需求文档链接、一句话需求描述。

这一步的作用是：不管用户输入什么，先统一转成后续流程能用的 km_links / 需求文档入口。

所以 yx-fsd-req-km-extractor 是入口路由器。它负责判断你给的是 FSD、KM，还是普通文字。如果是 FSD，它会去里面找 KM 链接；如果只是一句话，它可能会转成学城文档。这样后面所有流程都不用关心输入格式。

你可以理解成：

第一步不是开发，而是把需求入口标准化。

二、第二段：确定要改哪些仓库，并创建隔离开发环境
yx-repo-scope-bootstrap
 ↓
yx-using-git-worktrees

这一步做两件事。

第一，判断这个需求涉及哪些后端仓库：

active_repos
primary_repo

比如一个需求可能只改订单服务，也可能同时改订单服务、商品服务、client SDK、Thrift 接口仓库等。

第二，为每个仓库创建独立的 git worktree。

worktree 可以简单理解成：

在不污染主工作区的情况下，拉出一个专门给这个需求用的开发目录。

这样 AI 改代码的时候，不会直接把你当前本地分支搞乱。

这里有一个人工确认点：

确定 active_repos、primary_repo（人工确认）

也就是说，元析不会直接一上来就建分支改代码，而是先让人确认：“我理解这个需求要改这些仓库，对不对？”

三、第三段：先澄清业务，再分析原型
yx-pm-analyze-requirement
 ↓
yx-pm-prototype-validation

这里还没到技术方案，更没到写代码。

yx-pm-analyze-requirement 做的是业务需求澄清，产物是：

requirement.md

它主要回答：

这个需求到底要解决什么问题？

有哪些业务规则？

有哪些异常场景？

验收标准是什么？

哪些地方还不明确？

然后 yx-pm-prototype-validation 做原型交互分析，产物是：

prototype.md

它主要回答：

页面有哪些交互？

按钮点了之后应该发生什么？

前端会调用哪些后端接口？

有没有需要确认的 UI 行为？

所以这一段的核心是：

先搞清楚“要做什么”，再决定“怎么做”。

这也很像真实研发流程：不能 PRD 没看清、交互没确认，就直接开写代码。

四、第四段：设计技术方案，并按需补知识
yx-plan
 └─ yx-knowledge-learn
      ├─ yx-domain-knowledge
      ├─ yx-km-learn
      ├─ yx-middleware-learn
      └─ yx-infra-learn

yx-plan 是技术方案设计阶段，产物是：

design.md

它会根据前面的 requirement.md 和 prototype.md，再结合代码仓库，设计后端方案。

比如会考虑：

要不要新增接口？

改哪个 Service？

是否涉及 DB 表？

是否要接 Mafka / Crane / Lion / Squirrel / Cat？

是否要改 Thrift IDL？

是否涉及跨仓库调用？

有没有兼容性问题？

如果它发现自己不确定，就会调用 yx-knowledge-learn 去补知识。

这里的知识来源分几类：

Skill	作用
yx-domain-knowledge	补业务领域知识
yx-km-learn	读学城 / KM 文档
yx-middleware-learn	查中间件使用规范
yx-infra-learn	查 DB、Lion 等基础设施状态

这段可以理解成：

AI 在写方案之前，先主动查公司内部知识和项目上下文，避免瞎设计。

五、第五段：把技术方案拆成可执行任务
yx-writing-plans
 ↓
tasks.md

技术方案通常比较大，不能直接丢给 AI 一口气写。

所以 yx-writing-plans 会把 design.md 拆成多个具体任务，生成：

tasks.md

每个任务会尽量包含：

要改哪个文件；

要新增哪个类；

要写什么测试；

任务之间有什么依赖；

什么时候 commit；

哪些任务必须 TDD；

哪些任务只需要编译验证。

你可以理解成：

design.md 是“技术方案”，tasks.md 是“施工清单”。

六、第六段：如果有跨服务接口，先落接口契约
yx-interface-contract（可选）
 └─ yx-thrift-client-setup

这一步是可选的。

如果 tasks.md 里发现有跨服务接口，比如新增 Thrift RPC，那么元析不会直接写业务逻辑，而是先把接口契约落下来。

比如：

先写 .thrift IDL；

生成 Java client stub；

更新消费方依赖；

确保接口骨架能编译通过。

这样做的好处是：

先把跨仓库、跨服务的边界定清楚，避免后面多个任务互相卡住。

比如服务 A 要提供接口，服务 B 要调用接口。如果接口定义没先确定，两个仓库的开发都会混乱。

七、第七段：真正进入编码，但不是一次性乱写
编码执行：subagent-driven-development

这是整个流程最核心的部分。

它会读取 tasks.md，然后为每个任务创建 Todo。每个任务会进入一个循环：

implementer subagent 写代码
 ↓
先写测试
 ↓
再写实现
 ↓
测试通过
 ↓
自我 review
 ↓
commit
 ↓
spec-reviewer 审查
 ↓
code-quality-reviewer 审查
 ↓
任务完成

这里有几个关键点。

第一，编码不是主 Agent 自己直接写，而是派发给 implementer subagent。
也就是每个任务可以由一个专门的子 Agent 去做。

第二，强调 TDD：

先写测试 → 再写实现 → 测试通过

第三，写完不是直接算完成，而是要过两轮 review：

Review	看什么
spec-reviewer	是否符合需求和设计
code-quality-reviewer	代码质量是否达标

如果不符合，就回到 implementer 继续修。

所以这段不是：

“AI 写完代码就完事。”

而是：

AI 写代码 → AI 自测 → AI 自审 → AI review → AI 修复 → 再确认。

这就是它比普通 AI Coding 更工程化的地方。

八、第八段：全局自优化
yx-self-optimization

所有任务完成后，还会进入一次整体质量优化。

它会做类似这些事：

检查所有任务是否完成；

检查单元测试是否全绿；

做 lint 检查；

分析圈复杂度；

统一命名；

生成或补充集成测试；

必要时做自动修复。

这一步的作用是：

不是每个任务单独看没问题就行，还要站在整个 feature 维度做一次整体质量收口。

比如单个任务都能跑，但整体命名不一致、复杂度过高、部分边界没测到，这一步就会补。

九、第九段：开发收尾，决定是否 push / PR
yx-finishing-a-development-branch

这一步是人工确认点。

它通常会给几个选项，比如：

Push；

创建 PR；

保留本地；

丢弃变更。

如果用户选择 PR，也就是：

Option 2

后面就会并发触发两条线。

十、第十段：PR 后并发做两件事
SubAgent A: yx-bind-branch → yx-branch-deploy → yx-deploy-status
SubAgent B: yx-test-case-extract

这两条线是并发的。

A 线：部署链路
yx-bind-branch
 ↓
yx-branch-deploy
 ↓
yx-deploy-status

这条线负责：

把 feature 分支绑定到 FSD 需求；

在 FSD 平台创建研发交付；

部署到泳道环境；

检查部署状态。

也就是：

把代码部署到一个可测试的环境。

B 线：测试用例生成
yx-test-case-extract

这条线负责根据设计文档生成集成测试用例，比如：

哪些 RPC 接口受影响；

怎么构造请求参数；

预期返回什么；

前置状态和后置状态怎么验证。

产物一般是：

test-cases.json

也就是：

一边部署服务，一边准备测试用例，节省等待时间。

十一、第十一段：部署完成 + 用例生成完成后，做集成测试
yx-integration-test

这一步会读取 test-cases.json，然后通过 OCTO MCP 去远程调用泳道服务的 RPC 接口，做接口层自检。

它主要验证：

接口能不能调通；

核心业务逻辑是否符合预期；

返回结果是否正确；

数据状态是否符合设计；

失败时是环境问题、数据问题，还是代码问题。

文档里提到，如果用例失败，会先排查环境，再查日志，再查数据库，最后再决定怎么修；测试用例不能随便改，因为它代表验收标准。

所以这一步是：

不是只看代码能不能编译，而是看部署后的服务在测试环境里是否真的能跑通。

十二、最后：如果问题只是方法体变更，用热部署快速修复
yx-java-hot-deploy

如果集成测试失败，并且判断问题只是 Java 方法体内部的小修改，不涉及接口结构、类结构、依赖变更，那么可以用热部署。

热部署的作用是：

不重新完整部署服务，而是把改过的 Java 源文件或 class 文件快速推到远程服务器，让修复秒级生效。

如果是结构性变更，比如新增类、改接口、改依赖，那就不能热部署，要走完整部署。

总结成一句话

这个工作流可以概括成：

元析先把用户需求标准化，再确认仓库和开发环境；然后通过需求澄清、原型分析、技术方案和任务拆分，把需求变成可执行任务；接着用多个 SubAgent 按 TDD 编码、review 和优化；最后自动绑定分支、部署泳道、生成测试用例、执行集成测试，并在小范围问题下用热部署快速修复。

更简单地说：

需求输入
 → 需求理解
 → 仓库定位
 → 方案设计
 → 任务拆分
 → TDD 编码
 → 代码审查
 → 部署
 → 集成测试
 → 快速修复

它的本质就是：

把一个资深后端工程师从接需求到提测前的工作流程，拆成一组可被 AI Agent 调用的 Skill 链路。
