# Codex 项目操作手册


## 编码行为规范

### 1. 先想清楚再动手

不要假设。不要隐藏困惑。主动暴露权衡。

动手实现前：

- 明确说出你的假设。如果不确定，先问。
- 如果存在多种理解，列出来，不要自己悄悄选一个。
- 如果有更简单的方案，说出来，必要时反驳需求。
- 如果有不清楚的地方，停下来，说清楚哪里困惑，再问。

### 2. 简单优先

用最少的代码解决问题，不做任何推测性实现。

- 不加超出需求的功能。
- 单次使用的代码不做抽象。
- 不加没有被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景写错误处理。
- 如果写了 200 行但 50 行就够，重写。
- 问自己：“一个高级工程师会觉得这过度设计吗？”如果是，简化。

### 3. 外科手术式修改

只碰必须碰的地方，只清理自己制造的乱子。

修改已有代码时：

- 不“顺手优化”相邻代码、注释或格式。
- 不重构没有问题的东西。
- 保持已有风格，即使你会做得不一样。
- 发现无关死代码，提出来，不要擅自删除。

当你的改动产生孤儿代码时：

- 删除**你的改动**造成的无用 import、变量、函数。
- 不删除已有的死代码，除非被要求。
- 检验标准：每一行改动都应直接对应用户的需求。

### 4. 目标驱动执行

定义成功标准，循环直到验证通过。

把任务转化为可验证的目标：

- “加校验” -> “为非法输入写测试，再让它通过”
- “修 bug” -> “写一个能复现的测试，再让它通过”
- “重构 X” -> “确保重构前后测试都通过”

多步骤任务先说简短计划：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

明确的成功标准让你可以独立循环推进。模糊的标准（“让它能用”）会导致反复确认。

### 5. 测试边界

默认只做必要的代码逻辑测试，验证本次修改的关键路径即可。

- 优先跑与改动直接相关的单元测试、静态检查或最小复现脚本。
- 不主动扩大到全量手动验收、长时间浏览器验证或真实外部流程，除非用户明确要求。
- 如果某些验证更适合用户手动测试，先说明已完成的自动化验证和建议的手动测试步骤。
- 发现无关既有失败时，记录清楚失败项和原因，不把它混入当前任务范围。

---

这些规范生效的标志：diff 中不必要的改动变少、因过度复杂导致的返工变少、澄清性问题在实现前提出而不是出错后。
